## EXP-001 — 하단 탭 내비게이션 도입

**배포일**: 2026-03-31 (검수 1 제출)
**담당**: 단독  
**상태**: 🟡 검수 진행 중 (심사 대기, 데이터 수집 예정)
**커밋**: `e0da585`

### 배경

기존 앱은 단일 화면 구조로, 설정이나 보상 같은 기능이 상단 버튼 하나에 숨겨져 있었다.  
신규 기능(보상, 더보기)을 지속적으로 추가할 계획인데, 현재 구조로는 기능 발견율(Feature Discovery Rate)이 사실상 0에 가깝다.

### 가설

> **하단 탭 내비게이션을 도입하면, 앱 내 보상/설정 기능의 발견율이 높아지고  
> 결과적으로 세션당 체류 시간과 재방문율이 개선될 것이다.**

### Control (배포 전) 스냅샷

| 항목 | Control |
|------|---------|
| 네비게이션 | 단일 화면, 상단 우측 "설정" 버튼 1개 |
| 접근 가능 기능 | 알람 목록 + 설정 2개 |
| 보상 페이지 | 존재하지 않음 |
| 설정 접근 경로 | 알람 목록 → 상단 버튼 탭 |
| 현재 위치 표시 | 없음 |

### Treatment (배포 후) 스냅샷

| 항목 | Treatment |
|------|-----------|
| 네비게이션 | 하단 탭바 3탭 — 알람 목록 / 보상 / 더보기 |
| 접근 가능 기능 | 알람 목록 + 보상 + 설정 포함 더보기 |
| 보상 페이지 | 탭 1클릭으로 즉시 접근 |
| 설정 접근 경로 | 더보기 탭 직접 접근 |
| 현재 위치 표시 | 활성 탭 파란색 하이라이트 |

### 측정 지표

**Primary Metric**
- `/rewards` 페이지 방문율 (4월 MAU 대비 보상 페이지 방문 비율)

**Secondary Metrics**
- 탭 전환 발생 세션 비율 (홈 외 탭 접근 세션 %)
- 세션당 평균 페이지뷰 수 (이전 대비 증가 여부)
- 더보기 → 알림 설정 변경률

**Guardrail Metrics (악화 시 롤백 기준)**
- 알람 추가 완료율 — 탭 도입으로 혼란이 생길 경우 하락 가능
- 앱 이탈율 (첫 페이지 bounce)

### 성공 기준

| 지표 | 목표 |
|------|------|
| 보상 페이지 방문 세션 비율 | 배포 첫 주 **≥ 30%** |
| 더보기 탭 방문 세션 비율 | 배포 첫 주 **≥ 20%** |
| 알람 추가 완료율 | Control 대비 **유지 또는 개선** |

### 측정 방법

- 앱인토스 대시보드 페이지별 UV/PV 추적 (4월 데이터)
- 기준 비교: 3월 데이터 (설정 버튼 기간)
- 평가 시점: **배포 후 14일** (검수 통과 후 기산)

## EXP-002 — 10/30일 체크인 보상 시스템 도입

**배포일**: 2026-03-31 (검수 1 제출)
**담당**: 단독  
**상태**: 🟡 검수 진행 중 (심사 대기, 데이터 수집 예정)
**커밋**: `e0da585`
**의존성**: EXP-001 (보상 페이지 탭 접근 가능해야 함)

### 배경

3개월 MAU 데이터에서 확인된 핵심 문제: **재방문 동기 없음**.  
현재 앱은 알람을 등록한 뒤 매칭이 일어나지 않으면 재방문할 이유가 없다.  
매칭 대기 기간 동안 유저를 붙잡을 **중간 리텐션 루프**가 필요하다.

### 가설

> **일일 체크인 기반 보상 시스템을 도입하면, 앱 재방문의 명시적인 동기가 생겨  
> D7 이상 리텐션율이 개선되고 4월 MAU 하락폭이 줄어들 것이다.**

### Control (배포 전) 스냅샷

| 항목 | Control |
|------|---------|
| 재방문 인센티브 | 없음 |
| 알람 슬롯 획득 방법 | 유료 결제 (1,000원) 1가지 |
| 광고 제거 방법 | 없음 (항상 광고 노출) |
| 진행도 시각화 | 없음 |

### Treatment (배포 후) 스냅샷

| 항목 | Treatment |
|------|-----------|
| 재방문 인센티브 | 10일 / 30일 체크인 보상 2종 |
| 알람 슬롯 획득 방법 | 유료 결제 + **10일 체크인 무료 획득** |
| 광고 제거 방법 | **30일 체크인 달성 시 영구 제거** |
| 진행도 시각화 | `(N/10)`, `(N/30)` 배지 상시 노출 |
| "받기" 버튼 | 항상 노출, 조건 충족 시 활성화 |

### 체크인 로직

```
방문 날짜 배열 → localStorage 저장 (고유 날짜 기준)
10일 체크인: 누적 방문일 ÷ 10 = 완료 사이클 수 (반복 가능)
30일 체크인: 누적 방문일 ≥ 30 → 1회 수령 가능, api.claimAdFree() 호출 → DB adFree=true
```

### 측정 지표

**Primary Metric**
- **D7 리텐션율** — 앱 첫 방문 이후 7일 이내 재방문 비율 (앱인토스 대시보드)

**Secondary Metrics**
- 체크인 완료율 (10일 보상 수령 유저 비율)
- 평균 재방문 간격 (일)
- 30일 보상 도달율
- 보상 수령 이후 알람 추가율 (슬롯 +1 → 실제 알람 추가로 이어지는지)

**Guardrail Metrics**
- 인앱 결제(슬롯 구매) 전환율 — 무료 슬롯 지급으로 인한 감소 여부 모니터링

### 성공 기준

| 지표 | 목표 |
|------|------|
| D7 리텐션율 | 3월 대비 **+10%p 이상** 개선 |
| 10일 체크인 완료 유저 비율 | 4월 활성 유저 기준 **≥ 15%** |
| 4월 MAU 하락폭 | 3월(162) 대비 **50% 이내** (≥ 81) |

### 측정 방법

- 앱인토스 대시보드 리텐션 코호트 분석
- 기준 코호트: 2026-03 신규 유저 D7 리텐션
- 비교 코호트: 2026-04 신규 유저 D7 리텐션
- 평가 시점: **2026-05-01** (4월 MAU 집계 후)

## EXP-003 — 알람 추가 UX 개선 (광고 CTA + 하루 제한)

**배포일**: 2026-03-31 (검수 1 제출)
**담당**: 단독  
**상태**: 🟡 검수 진행 중
**커밋**: `e0da585`

### 배경

기존 알람 추가는 슬롯 초과 시 결제 팝업만 있었고, 알람 추가 자체의 마찰이 적었다.  
리워드 광고를 통해 무료 알람 추가를 허용하되, 하루 제한으로 남용을 방지한다.

### 가설

> **광고 시청 후 알람 추가 방식으로 전환하면, 유료 전환율 하락 없이  
> 알람 추가 완료율이 유지되고 광고 수익이 발생할 것이다.**

### Control (배포 전) 스냅샷

| 항목 | Control |
|------|---------|
| 알람 추가 방법 | 슬롯 내 무료, 슬롯 초과 시 결제 |
| 하루 추가 제한 | 없음 |
| CTA 버튼 텍스트 | "추가하기" |
| 광고 연동 | 없음 |
| 메시지 필드 | (미구현) |

### Treatment (배포 후) 스냅샷

| 항목 | Treatment |
|------|-----------|
| 알람 추가 방법 | 리워드 광고 시청 후 추가 (adFree 유저는 광고 없이 추가) |
| 하루 추가 제한 | 슬롯 수 × 2회 (기본 슬롯 2개 → 하루 4회) |
| CTA 버튼 텍스트 | "광고보고 추가하기" / "알람 추가하기" (adFree 분기) |
| 광고 연동 | AdMob 리워드 광고 (`ait.v2.live.3c9485e5e7974743`) |
| 메시지 필드 | 구현됨, 현재 숨김 (메시지 기능 검수 시 복원 예정) |

### 측정 지표

- 알람 추가 완료율 (광고 이탈율 포함)
- 광고 시청 완료율 (userEarnedReward 이벤트 기준)
- adFree 전환율 (30일 체크인과 연계)

### 성공 기준

| 지표 | 목표 |
|------|------|
| 알람 추가 완료율 | Control 대비 **-5%p 이내** (광고 마찰 허용 범위) |
| 광고 시청 완료율 | **≥ 70%** |

## EXP-004 — 연결 성공 화면 ID 표시 + 클립보드 복사

**배포일**: 2026-03-31 (검수 1 제출)
**담당**: 단독  
**상태**: 🟡 검수 진행 중
**커밋**: `e0da585`

### 배경

기존 연결 성공 화면은 "서로의 마음을 확인했어요" 메시지만 표시되고,  
실제로 연락을 취하려면 유저가 직접 인스타그램 앱을 열어 기억에 의존해야 했다.  
"인스타그램으로 연락하기" 외부 이동 버튼은 앱인토스 검수 정책으로 제거.

### 가설

> **연결 성공 화면에 양쪽 Instagram ID를 표시하고 클립보드 복사 기능을 제공하면,  
> 실제 오프앱 연락 전환율이 높아지고 서비스 핵심 가치가 완성될 것이다.**

### Control (배포 전) 스냅샷

| 항목 | Control |
|------|---------|
| ID 표시 | 없음 |
| 연락 방법 | 기억에 의존 (앱 내 안내 없음) |
| 외부 이동 버튼 | "인스타그램으로 연락하기" (검수 반려로 제거됨) |
| 복사 기능 | 없음 |

### Treatment (배포 후) 스냅샷

| 항목 | Treatment |
|------|-----------|
| ID 표시 | `from 본인ID ❤️ to 상대ID` 한 줄 표시 |
| 연락 방법 | 상대 ID 탭 → 클립보드 복사 → 토스트 "ID xxx 가 복사됐어요" |
| 외부 이동 버튼 | 제거 (정책 준수) |
| 복사 기능 | `navigator.clipboard.writeText()` |

### 측정 지표

- 연결 성공 화면 체류 시간 (복사 유도 효과)
- ID 복사 탭 발생율 (복사 이벤트 발생 세션 %)
- 연결 후 재방문율 (연락 성공 → 앱 재방문 동기 여부)

### 성공 기준

| 지표 | 목표 |
|------|------|
| ID 복사율 | 연결 성공 세션 대비 **≥ 50%** |

## EXP-005 — 인스타그램 인증 도입 + 초기 진입 안정화

**배포일**: 2026-04-09 (검수 2 승인 후 라이브 반영)
**담당**: 단독  
**상태**: ✅ 현재 출시 중  
**브랜치 흐름**: `review/2` → `main` 통합 (`8e78f84`)  
**빌드**: `019d7026-fc2b-78e2-9151-5f226b1e28d1`

### 배경

기존 서비스는 본인 인스타그램 계정을 검증할 수 없어 향후 메시지, 랭킹 노출, 신뢰 기반 기능 확장에 제약이 있었다.  
또한 검수/운영 과정에서 앱 시작 시 빈 화면처럼 보이는 구간이 확인되어, 인증 도입과 함께 초기 진입 안정화가 필요해졌다.

### 가설

> **인스타그램 인증을 도입하면 신뢰 가능한 계정 기반 기능 확장이 가능해지고,  
> 앱 시작 구간의 불확실성을 줄이면 인증 및 핵심 기능 진입 이탈도 함께 줄어들 것이다.**

- 메시지 신고/숨기기 기능
- 메시지 캐싱 (localStorage) 으로 재진입 시 즉시 표시
- 더보기: 읽지 않은 메시지 배지 표시 on/off 토글

### 측정 지표

**Primary**: 메시지 포함 알람 등록 비율 (전체 알람 대비)
**Secondary**: 메시지 페이지 방문 세션 비율, 이모지 반응율, 메시지 확인 후 알람 추가 전환
